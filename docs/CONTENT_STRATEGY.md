# Content Strategy

**Project:** CAMARA Network API Experience Centre

**Version:** 1.0

---

# Purpose

Content is the product.

Technology enables the experience.

Every page, demo and workflow should educate, inspire confidence and encourage exploration.

The Experience Centre should feel like a guided journey rather than a documentation website.

---

# Content Mission

Every visitor should leave with a better understanding of:

- the business challenge
- the customer journey
- the role of CAMARA Network APIs
- the implementation approach

Content should reduce complexity.

Never increase it.

---

# Solution Page Template

Every solution should contain the following sections.

## Hero

A concise statement describing the business outcome.

Example:

Passwordless Login

Eliminate OTP friction using trusted network authentication.

---

## Business Challenge

Describe the customer problem.

Focus on business language.

Avoid API terminology.

Questions to answer:

- What is happening today?
- Why is it difficult?
- Who is affected?

---

## Business Value

Explain measurable outcomes.

Examples:

- Reduce login friction
- Improve conversion
- Reduce fraud
- Improve trust
- Lower operational costs

Whenever possible describe outcomes rather than features.

---

## Customer Journey

Illustrate the user's journey.

Example:

Customer

↓

Launches banking app

↓

Authentication requested

↓

Number Verify

↓

Access granted

The visitor should immediately understand the flow.

---

## Interactive Demo

Allow visitors to experiment.

Examples:

Phone number

Transaction amount

Country

Risk profile

Run Demo

Interactive demonstrations should encourage exploration.

---

## Workflow

Explain how the solution works.

Example:

Application

↓

CAMARA APIs

↓

Business Logic

↓

Outcome

Use diagrams where appropriate.

---

## APIs Used

List all CAMARA APIs used.

For each API include:

- Purpose
- Why it is needed
- Link to API Explorer

Avoid lengthy descriptions.

---

## Architecture

Provide a simplified reference architecture.

Avoid unnecessary technical detail.

The diagram should support the explanation rather than replace it.

---

## Request / Response

Show representative API examples.

Prefer realistic data.

Highlight important fields.

Explain the response.

---

## Implementation Guidance

Help developers understand:

- prerequisites
- integration steps
- authentication
- expected behaviour

Avoid overwhelming detail.

Link to additional resources where appropriate.

---

## Related Solutions

Recommend additional demos.

Examples:

Passwordless Login

↓

Secure Payment

↓

Digital Onboarding

↓

SIM Swap Protection

Encourage continued exploration.

# Industry Pages

Industry pages should answer:

What problems does this industry face?

Which CAMARA APIs help?

Which demos should visitors explore?

Keep the focus on outcomes rather than technology.

---

# Writing Style

Write for intelligent readers.

Assume curiosity.

Never assume prior knowledge.

Use:

short paragraphs

plain language

active voice

clear headings

Avoid:

buzzwords

marketing clichés

unexplained acronyms

long technical explanations

---

# Tone of Voice

Professional

Helpful

Confident

Practical

Curious

Avoid:

sales language

hyperbole

fear-based messaging

---

# Naming Principles

Prefer outcomes.

Examples:

Passwordless Login

Secure Payment

Digital Onboarding

Fleet Visibility

Avoid API names as page titles.

API names belong inside the solution.

---

# Diagrams

Every diagram should simplify.

Never decorate.

Prefer:

workflow

sequence

architecture

decision trees

Avoid unnecessary complexity.

---

# Images

Every image must support learning.

Illustrations should:

show relationships

support explanations

reinforce workflows

Avoid decorative graphics.

---

# Interactive Elements

Interactions should answer questions.

"What happens if..."

Visitors should be encouraged to explore.

Every interaction should teach something.

---

# Accessibility

All content should be:

readable

keyboard accessible

screen-reader friendly

high contrast

responsive

Accessible content is good content.

---

# Consistency Checklist

Before publishing any new content ask:

Does this start with the customer problem?

Is the business value clear?

Does it explain the workflow?

Are the CAMARA APIs clearly identified?

Can a developer understand the implementation?

Does it link to related solutions?

Would a business stakeholder understand it?

---

# Future Expansion

The content model should support:

new APIs

new industries

new demos

new workflows

new regions

new languages

without changing the application architecture.

---

# Content Motto

**Every page should educate.**

**Every demo should inspire.**

**Every interaction should build confidence.**
